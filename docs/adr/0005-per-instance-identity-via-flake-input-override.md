# Per-instance identity via flake input override (--override-input on a non-flake device-id input)

Per-instance hostname, hostId, and sops file selection are resolved at Nix evaluation time by passing a DMI-derived hash into the flake as a non-flake input (`inputs.device-id` with `flake = false`). At provisioning time, a Python wrapper reads the target's DMI product UUID over SSH, hashes it to 8 hex chars, writes a `default.nix` returning `{ hostname, hostId }` into a temp dir, and invokes `nixos-anywhere --override-input device-id path:$tmpdir/device-<hash>`. The flake evaluates with the per-instance value baked in — no activation scripts, no runtime drift correction. A committed placeholder (`device-id/default.nix` returning `hostname = "unknown"`) keeps `flake.lock` resolution and `nix flake show` working without an override; a NixOS assertion in `profiles/base.nix` fails the build when the placeholder is in effect, so a forgotten override is caught at build time on the workstation rather than at boot on the target.

Sops files are keyed per-instance, role-prefixed: `secrets/<role>-<hash>.yaml` (e.g. `secrets/compute-a1b2c3d4.yaml`). `.sops.yaml` `creation_rules` match by role so new hosts of an existing role only need their public key appended to the role's `key_group`. The `first-*` k3s bootstrap variants are collapsed via a plain Nix helper (`hosts/mk-host.nix`) that takes `role` and `isFirstNode` as function arguments; the variants remain as distinct flake attrs because eliminating them entirely is a k3s topology change, not a module-organisation change.

**Rejected: runtime activation-script approach (prior version of this decision).** The initial design persisted the DMI UUID to `/var/lib/device-id` and used a NixOS activation script to set hostname at boot. This was rejected because the UUID was unavailable to Nix at eval time — anything baked into the closure that varied per instance couldn't depend on it, and the activation-script drift correction was a runtime hack for a build-time concern. The `--override-input` approach moves the value into eval where it belongs.

**Rejected: full dendritic pattern adoption.** Dendritic (flake-parts + import-tree + aspect model) was evaluated for this problem. Its factory pattern and auto-import mechanics address real duplication, but the plain-Nix layer (`mk-host.nix` helper + `readDir` host discovery) captures the same concrete benefit without adding two dependencies. Dendritic's distinctive value — the aspect model, where one file spans NixOS + home-manager + darwin for the same feature — has nothing to cut across in this headless-only, single-user, single-config-class cluster. Additionally, no canonical sops-nix pattern exists in the dendritic ecosystem; reference configs solve it four different ways. The cost (two new deps, a migration touching every file, conflicts with active worktrees) was not justified by the benefit.

**Rejected: fail-loudly without placeholder (Option 1).** A throwing placeholder would break `nix flake show`, IDE tooling (nil/nixd), and every dry-run build that doesn't supply an override. The placeholder+assertion combination (Option 3) preserves `flake.lock` resolution, `nix flake show`, and IDE eval while still catching forgotten overrides at build time. Note: `nix flake check` still fails under Option 3 because it forces `system.build.toplevel`, which trips the assertion — use `nix flake check --override-input device-id path:<real-dir>` to pass. This is accepted as a reasonable trade-off: the assertion's value (catching forgotten overrides at build time) outweighs the inconvenience of needing an override for `nix flake check`.
